# 作業提醒管理系統 (Homework Reminder Management System)

## 一、專題概述（Planning）

### 1.1 動機與背景
現在的學生課業繁忙，無論是國中生、高中生、甚至大學生，同時會有很多作業以及報告要繳交。為了避免在繁忙中忽略或搞混截止日期而逾期繳交，我們設計了這款能夠提醒大家作業和報告截止日的應用程式，以便使用者進行自主時間管理，降低課業焦慮。

### 1.2 目標使用者
*   **大學生**：特別需要自行規劃與管理多門課程的作業、課堂報告、期中期末考試時間。

### 1.3 專題目標
*   **目標一**：提供作業、考試與報告的集中管理功能（Dashboard）。
*   **目標二**：透過自動化的多管道提醒通知，大幅降低忘記截止日期的情況。
*   **目標三**：幫助使用者提早規劃時間，設置多段提醒，避免臨時抱佛腳。
*   **目標四**：打造簡單、高質感且直覺易操作的介面，提升日常使用的便利性。

## 二、需求分析（Requirements Analysis）

### 2.1 功能性需求（Functional Requirements）

| 功能編號 | 功能名稱 | 功能描述 |
| :--- | :--- | :--- |
| **F-01** | 截止日期追蹤與定時任務 | 使用者可以手動設定作業截止日期。系統會使用後台定時任務（Scheduler）輪詢，並在指定提醒時間點自動觸發提醒通知。 |
| **F-02** | 即時同步繳交狀態 | 提供一個快速勾選的檢查機制。若學生已完成作業並勾選已完成，系統應立即停止針對該項作業的後續所有提醒通知。 |
| **F-03** | 多方通訊軟體即時推送服務 | 整合外部 API（例如 LINE Notify、Gmail SMTP），確保提醒訊息能以推播或郵件方式直接送達學生的個人行動裝置。 |

### 2.2 非功能性需求（Non-functional Requirements）

| 類別 | 需求描述 |
| :--- | :--- |
| **效能** | 系統在新增、修改、查詢作業資料時，需要能快速回應，操作等待時間不超過三秒，並能同時處理多筆資料，確保使用流暢度。 |
| **安全性** | 系統需保護使用者個人資料與課業紀錄，登入帳號須具備密碼驗證與安全雜湊（Hash）機制，避免未授權使用者存取資料。 |
| **易用性** | 系統介面簡單清楚，功能分類明確。採用現代暗色玻璃擬態（Glassmorphism）風格，讓使用者快速上手，輕鬆完成新增作業、查看期限與設定提醒。 |

### 2.3 範圍限制（Out of Scope）
*   **本次不包含**：支付或金流功能
    *   *原因*：這是一個學術專題工具，不涉及商業收費模組，避免增加法律與資安上的風險。
*   **本次不包含**：與現有教學平台的自動 API 串接
    *   *原因*：各大教學平台的 API 權限申請流程繁瑣且容易更動，串接難度高。系統將以使用者自行輸入為主要資訊來源。
*   **本次不包含**：作業內容的批改與回饋系統
    *   *原因*：批改涉及文件註解、評分標準建立等複雜邏輯，偏離了「提醒」的核心。我們的系統將定位為「時程小幫手」而不是「教學管理平台」。

## 三、技術規劃（Technology Planning）

### 3.1 技術選型

| 層面 | 技術選擇 | 說明 |
| :--- | :--- | :--- |
| **前端** | HTML5 / CSS3 / Vanilla JavaScript | 純前端，無框架。使用現代 CSS 屬性（如 `backdrop-filter`）實現極致視覺的毛玻璃擬態，搭配 `Fetch API` 進行非同步請求。 |
| **後端** | Python + Flask | 課堂規範之後端框架，負責 RESTful API 路由分發與背景定時任務。 |
| **資料庫** | SQLite + SQLAlchemy | 輕量級關聯式資料庫，非常適合專題的資料持久化，無需額外架設伺服器。使用 SQLAlchemy ORM 加強開發安全性與效率。 |
| **其他工具**| APScheduler、LINE Notify API、smtplib (Gmail) | - **APScheduler**：用於在後台每分鐘輪詢是否有即將到期的作業並觸發通知。<br>- **LINE Notify** / **smtplib**：實現外接通訊推送服務。 |

- **溝通管道**：Discord / LINE

## 五、風險評估（Risk Assessment）

| 風險 | 發生可能性 | 影響程度 | 因應策略 |
| :--- | :--- | :--- | :--- |
| **外部 API 推送失敗（如憑證過期）** | 中 | 高 | 後端加入錯誤日誌處理，並於前端系統提供「即時通知歷史面板」作為內置的備援提醒管道。 |
| **後台任務因伺服器意外關閉而中斷** | 低 | 高 | 採用持久化任務記錄，於 Flask 重新啟動時自動檢查資料庫中尚未通知的作業，並重新啟動定時輪詢。 |
| **高頻率讀寫資料庫造成衝突** | 低 | 中 | 使用 SQLAlchemy 連線池與交易（Transaction）隔離級別，並在錯誤時自動進行 Rollback，保障資料一致性。 |
